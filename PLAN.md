# 1. 기획

## 1-1. 컨셉

"임금님 귀는 당나귀 귀" 설화 모티브 — 회사에서 하고 싶은 말을 대나무숲에 대고
외치면, 거친 표현이 정중하거나 웃긴 말투로 순화되어 나가고, 그 순간 숲에 있던
다른 사람들과 실시간으로 스쳐 지나가듯 채팅을 나눈다.

**핵심 재미 요소**
- 분노의 타이핑 → 순화 변환의 갭에서 오는 웃음 포인트
- 대나무숲 배경 + 외치기 애니메이션 + 사운드로 몰입감/타격감 제공
- 메시지가 저장되지 않고 그 순간 스쳐 지나감 (휘발성 컨셉)
- 직장인 공감 코드 (게시판 없이 가볍게 스쳐가는 채팅)

**타겟**
- 한국 직장인 (회사에서 못 하는 말을 가볍게 풀고 싶은 사람들)

**제외한 것**
- 게시판 / 영구 저장 DB 없음
- 로그인 없음 (완전 익명)

**레퍼런스**
- [온라인 담타](https://www.damta.world/) — 일상 행위를 인터랙티브 토이화 + 익명 채팅 결합 사례

## 1-2. 네이밍

| 사용자 노출 브랜드명 | 대나무숲 / Daenamusoop |

기술적으로 다루는 부분은 하이픈이 들어간 `daenamu-soop`을 일관되게 사용한다.

## 1-3. 개발 스코프 원칙
- 주말 프로젝트 규모, 단계적으로 확장 (Phase 1 → 2 → 3)

- Phase 1만으로도 완결된 결과물이 나오도록 설계 (채팅은 도전과제로 분리)

# 2. 설계

## 2-1. 전체 아키텍처

| 구성 요소 | 기술 | 배포처 | 비용 |
|---|---|---|---|
| 프론트엔드 | React + TypeScript (Vite) | Vercel 또는 Netlify | 무료 |
| 실시간 채팅 서버 | Node.js + Socket.io | Render (무료 티어) | 무료 |
| 순화 변환 로직 | 규칙 기반 치환 사전 (정규식) | 프론트 내 로직 | 무료 |
| 사운드/애니메이션 | Web Audio API, SVG/CSS 애니메이션 | 자체 제작 | 무료 |

**DB 사용 안 함.** Socket.io 서버 메모리에서만 접속자/메시지를 관리하고, 서버 재시작 시
모두 휘발됨. 원본(순화 전) 텍스트는 서버에 전송/저장하지 않고 클라이언트에서 변환 후
결과만 브로드캐스트하는 것을 원칙으로 한다 (프라이버시 + 컨셉 강화).

> ⚠️ Render 무료 티어는 15분간 요청이 없으면 슬립 모드로 전환되어, 이후 첫 접속자가
> 서버가 깨어날 때까지 30초~1분 정도 기다릴 수 있음. 취미 프로젝트 스코프에서는
> 감안하고 진행.

> Railway는 트라이얼($5) 종료 후 월 $1 크레딧만 남는 구조라 완전 무료 목표에는
> 부적합하다고 판단, Render로 결정함.

## 2-2. 데이터 흐름

```
[클라이언트 A]
  입력 문장 (원본, 로컬에만 존재)
       │
       ▼
  치환 사전 로직으로 순화 변환 (클라이언트 사이드)
       │
       ▼
  변환된 문장만 Socket.io emit
       │
       ▼
[Socket.io 서버] ── 메모리에서만 유지, 저장 없음
       │
       ▼ broadcast
[클라이언트 A, B, C, ... 전체 접속자]
  변환된 문장이 채팅 로그에 실시간으로 쌓임 (새로고침 시 사라짐)
```

## 2-3. 화면 설계

### 화면 1 — 메인 (대나무숲)
- 배경: 대나무숲 SVG/일러스트, 바람에 흔들리는 CSS 애니메이션
- 중앙: 텍스트 입력창 + "외치기" 버튼
- 배경음: 대나무 스치는 소리 (브라우저 자동재생 정책상 첫 클릭 시 재생 시작)
- 하단: 현재 접속자 수 표시 (예: "지금 이 숲에 47명이 숨어있어요")

### 화면 2 — 외치기 연출
- 제출 시 원본 문장은 잠깐 보였다가 사라짐
- 순화된 문장이 메아리처럼 퍼지는 텍스트 애니메이션 (fade + scale)
- 효과음 (메아리 사운드)

### 화면 3 — 채팅 패널 (사이드 or 하단 슬라이드업)
- 변환된 문장만 실시간으로 쌓임
- 접속 시 랜덤 익명 닉네임 자동 부여 (예: "성난다람쥐", "지친대나무")
- MVP에서는 자유 채팅 입력 없이, "외치기" 결과만 채팅창에 표시되는 방식으로 단순화

## 2-4. 순화 변환 로직 설계
- 단어 단위 치환보다 **문장 패턴 매칭**(정규식으로 어미/부사 처리) 우선
  - 예: "~같아", "~진짜", "~아 짜증나" 등 자주 쓰는 어미/부사 패턴 처리
- 초기 20~30개 패턴으로 시작, 완성도보다 "웃긴 변환 결과"를 우선순위로 둠
- 추후 확장 여지: 변환 강도 옵션(순한맛/매운맛), LLM API 연동 등은 Phase 3 이후 검토

# 3. 개발

## 3-1. 기술 스택 상세
- **프론트엔드**: React + TypeScript (Vite 기반)
- **애니메이션**: SVG + CSS keyframe (필요 시 Lottie 검토)
- **사운드**: Web Audio API
- **실시간 통신**: Socket.io (client + server)
- **서버**: Node.js (Express + Socket.io)
- **배포**
  - 프론트: Vercel / Netlify
  - 백엔드: Render 무료 티어

## 3-2. MVP 기능 리스트 (우선순위 순)

### Phase 1 — 순화 변환기 (채팅 없이 로컬 완결)
- [ ] Vite + React + TS 프로젝트 스캐폴딩 (`client/`)
- [ ] 텍스트 입력 + 제출 UI
- [ ] 치환 사전 기반 변환 로직 구현 (`transformDictionary.ts`)
- [ ] 대나무숲 배경 + 외치기 애니메이션
- [ ] 배경음 / 효과음 (Web Audio API)
- [ ] Vercel/Netlify에 프론트 1차 배포 및 동작 확인

### Phase 2 — 익명 채팅 (도전과제)
- [ ] `server/` Node.js + Socket.io 서버 스캐폴딩
- [ ] 로컬에서 클라이언트-서버 Socket 연결 테스트
- [ ] 접속 시 랜덤 닉네임 부여 로직
- [ ] "외치기" 결과를 전체 접속자에게 실시간 브로드캐스트
- [ ] 현재 접속자 수 표시 (Socket.io connection count)
- [ ] Render에 서버 배포 및 프론트-서버 연결

### Phase 3 — 다듬기 (여유 있으면)
- [ ] 변환 강도 옵션 (순한맛 / 매운맛 톤 선택)
- [ ] 임금님 귀 캐릭터가 욕 세기에 따라 반응하는 이스터에그
- [ ] 모바일 반응형 대응

## 3-3. 개발 순서 (Claude Code 작업 기준)

1. **프로젝트 스캐폴딩**: `client/`에 Vite + React + TS 세팅
2. **치환 사전 로직 구현 및 단위 테스트**: 입력 문장 → 순화 문장 변환 함수부터 독립적으로 개발 (UI 없이 함수 단위 테스트 먼저)
3. **대나무숲 UI + 애니메이션**: 정적 화면 먼저 완성 (채팅 없이 로컬 상태로만 동작)
4. **Phase 1 완결 후 배포 1차 확인** (Vercel에 프론트만 배포해서 정상 동작 체크)
5. **Socket.io 서버 스캐폴딩**: `server/`에서 로컬로 먼저 클라이언트-서버 연결 테스트
6. **Render 배포 + 프론트에서 서버 연결**
7. **접속자 수 / 닉네임 / 브로드캐스트 기능 순서로 추가**
8. **Phase 3 다듬기는 여유 될 때 진행**

## 3-4. 향후 고려 사항 (지금은 스코프 아님)
- LLM API 연동을 통한 순화 톤 다양화 (비용 발생하므로 무료 목표와 별도 검토 필요)
- 변환 사전 확장 및 커뮤니티 피드백 반영
- 접속자 통계, 시간대별 트래픽 분석 (단, DB 없는 원칙과 상충하므로 별도 설계 필요)
